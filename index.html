Executive Summary

The headline: A licensed M365 Copilot Chat seat — already paid for as a flat monthly cost — was used to perform what is normally a multi-day senior-engineer code review on a production Azure application. The Researcher agent delivered an architecture blueprint, data-flow reconstruction, security audit, and prioritized remediation plan grounded in actual source code, with concrete production-telemetry validation. No metered AI service was billed for any of it.

The key insight is that M365 Copilot Chat is an unmetered reasoning surface. When you can hand it the right artifacts in the right order, it will perform deep, deterministic code analysis on real applications — the same kind of analysis that, on a token-metered platform, would carry a meaningful per-engagement cost. This document describes the exact workflow used to do that on a real application (the SendGrid → Splunk Azure Functions pipeline), so the approach can be replicated for any codebase.

The Problem

Modern code reviews of unfamiliar production systems are expensive in two different ways:

  • Engineer time — reading every config file, every infra module, every C# class, every pipeline YAML and reconstructing the architecture by hand is a multi-day exercise per system.
  • AI credit spend — metered AI coding assistants charge per token. Pointing them at a full repository to "go figure it out" is the most expensive failure mode they have: every wrong turn, every clarifying re-read, every dead-end exploration burns credits.

Worse, naive file uploads to AI chat surfaces routinely break on real-world source: backticks, dollar-variables, embedded quotes, and special tokens get mangled by the chat's stream parser, silently corrupting the content the AI ends up reasoning over.

The workflow below solves both problems simultaneously: it produces a clean, faithful ingestion of the entire codebase, and it routes all of the reasoning to the flat-rate Copilot Chat surface.

How the Workflow Operates

Local Export (PowerShell GUI)

A WPF-based exporter scans the source tree, honors .gitignore / .dockerignore / .terraformignore and similar rules, optionally includes binaries, and emits one Base64-encoded .txt per file into a flat output directory.

Primer Artifacts Generated

Alongside the encoded files, the exporter writes MANIFEST.txt (full inventory with paths, sizes, MD5 hashes, dependency hints) and _CONTEXT.md (a primer that orients the AI to the project shape and ingestion protocol).

Researcher Agent Plans Phases

Only the two primer files are uploaded first. The agent reads them, infers project scope, and proposes a conservative multi-phase upload plan that fits within message and context constraints — preventing timeouts and partial parses.

Phased Ingestion & Synthesis

Encoded files are uploaded in the agent-defined order. After each phase, the agent decodes, verifies MD5 integrity, and reports findings. Once all phases complete, it produces a single end-to-end report.

Phased Ingestion Model — Why It Matters

Large codebases do not fit cleanly into a single AI chat turn. Even when individual files are small, the cumulative context budget is finite, and the model's attention degrades when too much unrelated material is loaded at once. The phased ingestion approach treats the codebase like a build pipeline: the agent only sees what it needs, in the order that compounds understanding.

Example: SendGrid → Splunk (27 files, 4 phases)

PHASE 1 · 13 FILES

Configuration & Manifests

App settings (dev / local / prod), host configuration, solution & project descriptors, deploy profiles, CI/CD pipeline. Establishes the runtime contract everything else binds to.

PHASE 2 · 9 FILES

Foundational Modules

Terraform root modules and variables, environment-specific tfvars and remote-state backend configs, dependency-injection bootstrap, and the core domain model.

PHASE 3 · 1 FILE

Application Entry Point

The trigger function that orchestrates the runtime workflow. Decoded last among code files so the agent already has full context for every method it references.

PHASE 4 · 4 FILES

Documentation & Diagrams

README, ignore rules, the editable diagram source, and the rendered architecture image — preserved for documentation parity, deliberately sequenced last because the code already established ground truth.

Why phasing prevents failure: Each phase is small enough to decode and verify integrity (MD5) before the next is sent. If anything fails — a parse error, a hash mismatch, a context overflow — the failure is isolated to one bounded batch. The work in prior phases is preserved, and remediation is surgical rather than starting over.

Outcomes & Tangible Deliverables

For the SendGrid → Splunk pipeline specifically, the Researcher agent produced the following — all grounded in decoded source, all cross-validated against production telemetry:

📐 Architecture Blueprint

End-to-end reconstructed diagram of the two-function Azure pipeline, the Table Storage handoff between them, the Splunk HEC sink, and the Key Vault–resolved secret flow.

📦 Full Inventory

Subscription topology across workload, infrastructure, and security accounts; resource-naming conventions; environment differences (dev S1 → prod P2v2 plans).

🔁 Data Workflow Summary

Step-by-step trace from inbound SendGrid webhook through manual JWT validation, Table persistence, timer-driven batch retrieval, and outbound HEC POST.

🚨 Concrete Bug Discoveries

A silently-broken cron expression that collapses to "every minute" rather than the intended interval; an anonymous HTTP intake whose only defense is a hardcoded Microsoft Graph audience; an all-or-nothing batch with no idempotency.

🛡️ Security & Configuration Notes

Managed-identity-to-Key-Vault flow validated; HEC token scheme identified; VNet egress assumptions surfaced; a referenced-but-missing /api/health endpoint flagged.

🎯 Prioritized Remediation

Nine fixes tagged P0 / P1 / P2 with specific code-level rationale, ranging from the cron correction and idempotency marker to dead-letter handling and consolidating duplicate DTOs.

When This Approach Is Most Effective

✅ Wins big when…

  • Inheriting an unfamiliar production application and needing an architecture brief fast.
  • Auditing a multi-environment Azure / IaC deployment for drift, risk, or compliance gaps.
  • Reconstructing an end-to-end data flow that spans code, infrastructure, and CI/CD.
  • Preparing a remediation plan that engineering leadership and senior ICs both need to trust.
  • Producing executive-shareable artifacts (architecture diagrams, risk registers, prioritized backlogs) on a single chat-flat-rate budget.

⚠️ Less effective when…

  • The task is a single-file, mechanical edit — the phasing overhead outweighs the benefit.
  • The codebase requires runtime execution or live debugging to understand (static analysis only goes so far).
  • Source materials cannot leave a tightly-controlled environment that doesn't include M365.
  • Wall-clock urgency trumps thoroughness — phased ingestion takes deliberate turns, not seconds.
